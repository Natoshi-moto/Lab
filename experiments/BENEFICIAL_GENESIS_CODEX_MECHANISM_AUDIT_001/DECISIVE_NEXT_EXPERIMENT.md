# Decisive next experiment

Run a preregistered, no-token/non-transferable-receipt product-function challenge before any additional token simulation.

The operator must specify a finite acceptance table for required functions: charity payment, migration admission, recognition, payment medium, fees, staking/security bond, governance, bootstrap ownership, and store-of-value. Implement the smallest non-transferable receipt path and compare it against the transferable proposal while holding every marked-required function constant. Reject transferability unless at least one required function fails without it and cannot be met by a separable ledger unit.

Pass criterion: a named required function with an executable acceptance test uniquely requires transferable allocation. Fail criterion: receipt/no-token meets all required functions. Empirical and legal questions remain external gates regardless.
