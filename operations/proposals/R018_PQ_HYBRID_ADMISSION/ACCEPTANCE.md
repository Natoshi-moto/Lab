# R018 acceptance and non-promotion criteria

R018 may carry only
`DEMONSTRATED_FIXED_PQ_ADMISSION_AND_UNCHANGED_R016_ALL_OF_GATE` when all gates
below pass on one exact proposal head.

1. Canonical R016, the R017 draft baseline, and the predecessor root dependency
   manifest and lock remain byte-exact.
2. Exact Node 24.14.0 and pinned Noble 0.6.1 are installed from the isolated
   R018 lock, whose registry integrity and installed entrypoint are rechecked.
3. The public deterministic fixture rebuilds byte-for-byte.
4. Native and cold verifiers emit the frozen byte-identical admission report.
5. Each ML-DSA-65 implementation verifies a signature generated by the other.
6. The exact R016 event passes the unchanged R016 kernel only after both PQ
   prechecks pass.
7. The R016-only baseline accepts that event while the hybrid profile rejects
   it when PQ authorization is absent.
8. Event/signature mutation, policy/controller/epoch replay, algorithm
   downgrade, unknown override fields, self-consistent fallback weakening, and
   key substitution fail without a passing report.
9. The primary gate contains no signing, private-key, key-generation,
   policy-update, override, or promotion surface.
10. All reports retain `status_authority: NONE` and every nonclaim remains
    explicit.

The claim must remain local to the R018 profile. It may not be restated as
network-wide mandatory PQ enforcement while legacy R016 remains directly
callable and no consensus/deployment rule exists.

Passing these gates does not authorize merge into canonical state. R018 remains
`PROPOSE_ONLY`; promotion requires a separate human decision after R017's own
status is resolved and after the missing key-lifecycle problem is explicitly
addressed or accepted as an experiment boundary.
