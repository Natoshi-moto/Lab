02 — What Nexus OS actually is

Nexus OS is a single HTML file (nexus-os-patched.html) that acts like a tiny operating system in your browser. It creates sandboxed <iframe> windows called blocks. Each block is its own HTML file — Verse Studio, the Wallet, NexusDB, IWW — running inside a sandbox with no access to each other's code or DOM. They can only talk through a controlled channel: the kernel. The kernel decides who can talk to who, routes messages between blocks, and kills blocks that misbehave. This is why you need to run everything through Nexus OS for cross-block features — outside of it, blocks can't see each other at all.

03 — The IPC handshake

Every block goes through the same boot sequence when Nexus OS loads it. Here's what happens, in order:

1
BOOT — The kernel sends a BOOT message to the block's iframe with a MessagePort (a private two-way channel). This is the only time the block hears from the kernel on the main window.message channel. Everything after this goes through the port.
2
DECLARE — The block sends back a DECLARE with its manifest: which channels it will emit (publish to) and which it wants to consume (subscribe to). IWW declares: emits iww.admission, consumes db.result.
3
MOUNT_CHALLENGE / MOUNT_ACK — The kernel sends a random nonce. The block echoes it back. This proves the block is alive and responding correctly before being let onto the bus.
4
MOUNTED — The kernel confirms the block is mounted. Now the block sends SUB messages for each channel it wants to actively receive. Important: declaring a channel in your manifest is an allowlist — it says "I'm allowed to receive this." Sending SUB is what actually subscribes you. Without SUB, messages on that channel are delivered to nobody.
5
PING / PONG — The kernel periodically pings every mounted block. The block must reply with PONG containing the same nonce. Fail enough pings and the kernel evicts the block. This keeps zombie blocks from clogging the system.
// What IWW's boot code looks like
window.addEventListener("message", e => {
  if (!e.data || e.data.type !== "BOOT") return;
  kernelPort = e.ports[0];

  kernelPort.postMessage({
    type: "DECLARE",
    manifest: {
      emits:    ["iww.admission", "system.block_ready"],
      consumes: ["db.result"]
    },
    app: { title: "I Was Wrong", icon: "✗" }
  });

  kernelPort.onmessage = handleKernelMsg;
});

function handleKernelMsg(ev) {
  const msg = ev.data;
  if (msg.type === "MOUNT_CHALLENGE")
    kernelPort.postMessage({ type: "MOUNT_ACK", nonce: msg.nonce });
  else if (msg.type === "MOUNTED") {
    kernelPort.postMessage({ type: "SUB", channel: "db.result" });
    // now we're live — load from DB
  } else if (msg.type === "PING")
    kernelPort.postMessage({ type: "PONG", nonce: msg.nonce });
  else if (msg.type === "MSG")
    handleIncomingMessage(msg);
}
The crucial bug to avoid (BUG-10 in NexusDB docs): If your block declares a channel in consumes but never sends SUB, it receives nothing and there's no error message. Silent failure. Always send SUB for every channel you actually want to receive after MOUNTED.

04 — How NexusDB stores things

NexusDB is a block (nexus-db.html) that wraps the browser's IndexedDB with a quasi-SQL interface over IPC. It has one main table: social_events. IWW stores its entries there using a Nostr-style kind field to distinguish its events from the rest.

Sending a query to NexusDB

You don't call NexusDB directly. You send a MSG on the db.query channel through the kernel. NexusDB is subscribed to that channel. It processes your query and emits the result on db.result, which you're subscribed to.

// Store an entry in NexusDB
kernelPort.postMessage({
  type: "MSG",
  channel: "db.query",
  payload: {
    query: "INSERT INTO social_events",
    params: {
      id:         entry.id,
      kind:       30078,        // IWW event kind
      pubkey:     entry.pubKeyHex,
      content:    JSON.stringify(entry),
      created_at: Math.floor(Date.now() / 1000),  // ← UNIX SECONDS not ms!
      nostr_sig:  entry.nostrSig
    },
    appId: "iww",
    _reqId: "iww_12345"  // match this to the db.result response
  }
});

// Read entries back
kernelPort.postMessage({
  type: "MSG",
  channel: "db.query",
  payload: {
    query:   "SELECT * FROM social_events",
    filters: { kind: 30078 },
    sort:    { column: "created_at", direction: "DESC" },
    appId:   "iww",
    _reqId:  "iww_12346"
  }
});
The timestamp bug (BUG-01): NexusDB stores created_at in Unix seconds (Nostr standard: ~1,745,000,000). If you pass Date.now() directly (milliseconds: ~1,745,000,000,000), the garbage collector will think the event is 1000 years old and delete it within 5 minutes. Always use Math.floor(Date.now() / 1000).

NexusDB has a namespace guard: blocks can only write to social_events, config, or tables starting with their appId. IWW uses social_events and filters by kind: 30078.

05 — How Nostr signing works in IWW

Every admission gets signed as a Nostr event (NIP-01). This is what makes the "cannot delete" claim technically honest — the event ID is a cryptographic hash of the content, and your Schnorr signature over that ID proves you authorised it.

The key pair

On first load, IWW generates a fresh secp256k1 private key using crypto.getRandomValues() and derives the public key using @noble/curves. Both are stored in localStorage. You can import your existing Nostr nsec instead.

The event structure (NIP-01)

{
  kind:       30078,          // parameterized replaceable, application-specific
  pubkey:     "hex pubkey",
  created_at: 1745000000,    // unix seconds
  tags: [
    ["d",        "admission-id"],    // unique identifier
    ["t",        "iww"],              // topic tag
    ["t",        "proof-of-error"],
    ["domain",   "technology"],
    ["score",    "81"],
    ["defended", "true"]
  ],
  content:    "JSON string of full admission data",
  id:         "sha256 of canonical serialisation",
  sig:        "schnorr signature over id"
}

The canonical serialisation (NIP-01)

JSON.stringify([
  0,               // reserved
  pubkey,           // hex
  created_at,       // unix seconds
  kind,             // 30078
  tags,             // array of arrays
  content           // string
])

The id is the SHA-256 of this string. The signature is a Schnorr signature (not ECDSA) over the id bytes, using the private key. Because SHA-256 is available in the Web Crypto API natively, the only external library needed is @noble/curves for the Schnorr step.

What "signed" means in practice right now: The event is signed and the ID and signature are stored locally. It is not published to a Nostr relay yet — that step is future (see section 08). But the signature is valid and verifiable by any Nostr client that sees the event.

06 — How the wallet ties in

This is the part that connects IWW to the rest of Nexus. Here is the intended flow, and what's real vs future:

The intended connection

When you submit an admission, IWW emits an iww.admission event to the kernel. The Wallet block (if running) can subscribe to iww.admission and offer to stake ATT tokens on your public position. The admission's Nostr event ID becomes the terms_raw field in the wallet's LOCK operation — the thing you're staking on.

// What the Wallet would receive when IWW emits
{
  id:           "u_1745000000_abc123",
  domain:       "technology",
  score:        81,
  pubkey:       "hex pubkey of the admitter",
  nostrEventId: "sha256 of the Nostr event"
}

// Wallet would then offer: "Stake on this position?"
// LOCK: terms_raw = "nostr:event:<nostrEventId>"
// The stake binds to the immutable Nostr event

The four wallet conservation rules (reminder)

RuleWhat it enforcesWhere it lives
Rule 1No UTXO can be consumed twiceApp-level (Wallet block)
Rule 2Only the authorised key can spendApp-level
Rule 3Signature must be validApp-level
Rule 4Inputs = Outputs (no creation or destruction)App-level
Current honest state: All four rules live in the Wallet block's own JavaScript. The kernel does not enforce them. A malicious block could theoretically send crafted wallet messages. This is a known gap (Grok's report confirmed it). It doesn't affect IWW directly — IWW just emits events; it doesn't touch the wallet — but it's worth knowing.

07 — How to run IWW in Nexus OS today

Requirement: You must serve the files from a local web server, not by double-clicking. Chrome silently drops MessagePort transfers to sandboxed iframes on file://.
1
Get a local server running. In the folder where your HTML files live, run:
python3 -m http.server 8080
Then open http://localhost:8080/nexus-os-patched.html
2
Add IWW to the Nexus app shelf. In Nexus OS, right-click the desktop → Add Block → enter the URL http://localhost:8080/i-was-wrong-v2.html. Give it the title "I Was Wrong" and icon ✗.
3
Also load NexusDB. IWW needs NexusDB to store entries. Add nexus-db.html as a block the same way. It will auto-boot and connect — you'll see its green monospace log in its window.
4
Open IWW. Click it from the shelf. You should see the Nexus OS indicator in the masthead: ⬡ Nexus OS. This means it detected it's running inside the kernel and is using NexusDB instead of localStorage.
5
Submit an admission. Your Nostr identity is auto-generated on first load. Submit your first real error. It will be signed with your key and stored in NexusDB, and an iww.admission event will be emitted to the kernel (currently goes nowhere, but it's there for when the Wallet block subscribes to it).
Crypto library note: IWW tries to load @noble/curves from https://esm.sh. If you're offline, signing falls back gracefully — entries are still saved, just marked unsigned. To guarantee signing works offline, run python3 setup_libs.py from your project folder first (this downloads the library locally for the Social block, same library IWW needs).

The two things that will most improve this, in order

1. Export/import signed bundle. Right now all your admissions die if you move to a new origin. A signed JSON export (all entries, signed by your Nostr key) that can be imported anywhere makes the "portable quine" claim honest. This is maybe 80 lines of code.

2. Nostr relay publish. Add a relay URL field in settings. On submit, open a WebSocket to the relay and send the signed event. Instantly your admission is on the public Nostr network, verifiable by anyone. This is maybe 40 lines of code.

Everything else — wallet staking, CPL integration, the 64 agents — builds on these two foundations being solid.
